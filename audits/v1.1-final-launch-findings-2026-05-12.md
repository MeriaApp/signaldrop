# SignalDrop v1.1 — pre-launch adversarial pass

**Date:** 2026-05-12
**Source:** `general-purpose` (Sonnet 4.6) agent + primary-source fact-check via Read.
**Bar:** "Would Apple ship this on their main product?"
**Verdict:** 3 real bugs + 1 design-judgment ask + 1 smoothing. NOT submit-ready as-is.

## Findings

### F1 — Wi-Fi 6E (6 GHz) networks misclassified as 2.4 GHz · SHOWSTOPPER

**Where:** `Sources/SignalDrop/NetworkScanner.swift:34-41`

```swift
static func from(channel: Int) -> ChannelBand {
    switch channel {
    case 1...14: return .band2_4GHz
    case 36...177: return .band5GHz
    case 1...233 where channel >= 1: return .band6GHz   // UNREACHABLE
    default: return .unknown
    }
}
```

The third case `1...233` is unreachable because `1...14` matches every 6 GHz channel in the low range first. Channel-number alone is also ambiguous — 6 GHz uses channels 1, 5, 9, 13... which overlap 2.4 GHz numbering.

**User-facing failure:** Mac connected to a Wi-Fi 6E AP. Scanner table shows "2.4 GHz, 20 MHz" for it. Band filter dropdown "6 GHz" returns zero networks. **Marketed v1.1 feature broken on every Apple Silicon laptop from 2023+.**

**Fix:** Use `CWWLANChannel.channelBand` (returns `.band2GHz` / `.band5GHz` / `.band6GHz` from Apple directly). Drop the channel-number heuristic. 8 LOC change in `makeScannedNetwork`.

**Confidence:** High.

### F2 — Phantom drops wreck the History grade · SHOWSTOPPER

**Where:** `Sources/SignalDrop/SignalDropApp.swift:221` + `ConnectionHistoryReport.swift:213-243` + `scoreFromOutages:285-289`

Phantom-drop suppression in `handleEvent` only suppresses the **notification**. Line 221 logs the disconnect to the event database **before** the deferral timer evaluates. The History tab's `reconstructOutages` then walks every disconnect→connect pair regardless of duration, and `scoreFromOutages` penalizes `-8 per outage + -3 per minute of downtime`.

**User-facing failure:** User in a coworking space roams between APs 12 times in a day (each <5s). Gets the silence the app promised. Opens History tab → Grade F, 12 outages, score 4/100. Feels cheated. The two parts of v1.1's marketing contradict each other inside the same app.

**Fix:** Filter `reconstructOutages` results by `notificationSettings.minDisconnectDurationSeconds` before passing to `scoreFromOutages` and the per-network rollup. Phantom drops stay in the raw event log (data integrity) but vanish from the grade math + outage count + per-network breakdown. Surface a "X brief roams filtered" footnote so the user understands.

**Confidence:** High.

### F3 — Onboarding tips reference removed UI · HIGH-VALUE POLISH

**Where:** `Sources/SignalDrop/OnboardingView.swift:165, 167`

```swift
tip("Click the WiFi-arc icon in your menu bar for live status.")
tip("Use Generate ISP Report to export downtime data for support chats.")
```

Both tips are stale after v1.1 polish Round 1:
- The "WiFi-arc icon" was `dot.radiowaves.left.and.right` in v1.0. v1.1 uses state-driven `wifi` / `wifi.exclamationmark` / `wifi.slash` / `lock.icloud`. "WiFi-arc" doesn't describe what's on screen.
- "Generate ISP Report" menu item was removed in commit `0ee67eb` (Round 1 menu cleanup). PDF export now lives in Network Insights → Connection History.

**User-facing failure:** First-time user finishes onboarding, reads three tips, the third tells them how to use a button that doesn't exist. App Review reads onboarding flows — this is also a "metadata vs binary mismatch" smell for a Guideline 2.3.1 reviewer.

**Fix:** Rewrite the two tips. ~2 LOC change.

**Confidence:** High.

### F4 — Quiet Hours bypassed for disconnects · DESIGN-JUDGMENT ASK

**Where:** `Sources/SignalDrop/SignalDropApp.swift:280-281` + `SettingsView.swift:100-101`

```swift
let isCritical = event.type == .disconnected || event.type == .internetLost
guard notificationSettings.shouldNotify(for: event.type, isCritical: isCritical) else { return }
```

Disconnects and internet-lost are hardcoded `isCritical: true`, which bypasses Quiet Hours AND fires `.defaultCritical` notification sound. The Settings footer explicitly advertises this: *"Disconnects and internet-lost still fire because they're the events you actually need to act on."*

**User-facing failure:** Cable modem resets at 3am. WiFi drops 20s, reconnects. Disconnect notification fires with critical sound at 3am despite Quiet Hours 10pm→7am being on. User wakes up. 1-star review: "Has Quiet Hours but ignores it."

The current behavior is INTENTIONAL — Jesse may have shipped it deliberately as "outages are too important to silence." The agent disagrees. Both views are defensible.

**Recommended fix (if Jesse agrees):** Quiet Hours suppresses *all* notifications by default. Add an opt-in "Override quiet hours for outages longer than N minutes" toggle for users who genuinely want the wake-up. Decouple `.defaultCritical` sound from "bypasses quiet hours."

**Confidence:** High that the bug exists. Whether to fix is Jesse's call.

### F5 — 5s notification throttle suppresses back-to-back drops · SMOOTHING

**Where:** `Sources/SignalDrop/SignalDropApp.swift:48-59, 282`

```swift
private let throttleIntervals: [WiFiEventType: TimeInterval] = [
    .disconnected: 5,
    .connected: 5,
    ...
]
```

The 5s throttle was added when phantom-drop suppression didn't exist. Now that phantom-drop deferral handles the spam case, the additional 5s throttle on `.disconnected` will silently swallow the second of two genuinely-separate outages within 5s of each other (the exact pattern of a thrashing cable modem).

**Fix:** Drop the throttle on `.disconnected` and `.connected` entirely. Keep on `.signalDegraded` / `.signalRecovered` / weaker events where flapping is still possible.

**Confidence:** Medium — depends on real-world ISP failure patterns. Low risk to ship as-is.
